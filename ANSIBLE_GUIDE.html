💡 What is Ansible? (Plain English)

Think of Ansible like a "remote control" for servers

Normally, to set up a server you would log in, run commands manually, install software, edit config files, etc. Ansible lets you write those steps once in a file (called a playbook), and then run them automatically on any number of servers — all from your own laptop, without ever logging in manually.

Key concepts you need to know:

TermWhat it means (simply)
Control NodeYour laptop / PC — the machine running Ansible
Managed NodesThe remote servers Ansible configures (your 3 Linux VMs)
InventoryA list of your servers (with their IP addresses)
PlaybookA recipe file listing what to install and configure
RoleA reusable bundle of tasks (e.g., "install MongoDB")
VaultAn encrypted file where Ansible stores passwords safely
SSHHow Ansible connects to your servers (like a secure tunnel)
✅ Good news! Ansible is agentless — you do NOT install anything special on your servers. Ansible connects via SSH (which Linux servers already have) and runs commands remotely.

✅ Before You Start — Prerequisites

What you need:

  • A Linux or macOS laptop/PC to run Ansible from (Windows users: use WSL2 — Windows Subsystem for Linux)
  • 3 Linux servers (Rocky Linux 8/9 or AlmaLinux recommended — these match the project's RHEL/CentOS setup). Can be VMs, cloud VMs (AWS, Azure, DigitalOcean), or VirtualBox VMs on your laptop
  • SSH access to all 3 servers (you need root or sudo access)
  • Python 3.8+ installed on your laptop (usually already there on Linux/macOS)
  • Maven installed on your laptop (to build the Java WAR file)
  • Basic familiarity with a terminal / command line
💡 Tip — Using VirtualBox for practice: If you don't have cloud servers, you can create 3 VirtualBox VMs on your own computer. Set them to "Host-Only Adapter" networking with IPs 192.168.56.10, .11, .12 — these match the dev inventory exactly.
⚠️ Operating System: This project is written for RHEL/Rocky Linux/AlmaLinux/CentOS (it uses yum and firewalld). It will NOT work on Ubuntu/Debian without modifications to the playbooks.

1 Install Ansible on Your Computer

Run these commands on your laptop/PC (your control node):

On macOS:

# Install Homebrew first if you don't have it:
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# Then install Ansible:
brew install ansible

On Linux (Ubuntu/Debian):

sudo apt update
sudo apt install -y python3-pip
pip3 install ansible --user

On Linux (RHEL/Rocky/AlmaLinux):

sudo dnf install -y epel-release
sudo dnf install -y ansible

On Windows (WSL2):

# Open WSL2 terminal, then:
sudo apt update && sudo apt install -y python3-pip
pip3 install ansible --user

Verify the installation:

ansible --version

You should see output like ansible [core 2.16.x]. ✅

2 Prepare Three Linux Servers

You need 3 servers running Rocky Linux 9 (or AlmaLinux 9). For each server:

  • Set a static IP address (or note the IP if it's assigned by DHCP)
  • Make sure SSH is running (systemctl status sshd)
  • Have root access (or a user with sudo)
📝 Note down your IP addresses!
You will need them in Step 4. The default in this project is:
  • App server: 192.168.56.10
  • DB server:  192.168.56.11
  • CI server:  192.168.56.12
If your servers have different IPs, no problem — you'll update the inventory in Step 4.

On each server, create the "deploy" user that Ansible will use:

# Run this on EACH server (log in as root first):
useradd -m -s /bin/bash -G wheel deploy

# Allow the "wheel" group to use sudo without a password:
echo '%wheel ALL=(ALL) NOPASSWD: ALL' | tee /etc/sudoers.d/wheel-nopasswd

# Verify:
su - deploy
sudo whoami    # Should print: root
✅ Why "deploy" user? The ansible.cfg file sets remote_user = deploy, meaning Ansible will always connect as the "deploy" user. This is a security best practice — Ansible never logs in as root directly.

3 Set Up SSH Key Authentication

Ansible connects to your servers via SSH. Instead of typing a password every time, we use SSH keys (a pair of digital "locks and keys").

3a. Generate an SSH key on your laptop (if you don't have one):

ssh-keygen -t ed25519 -C "ansible-cmdb"
# Press Enter to accept defaults (no passphrase needed for automation)

This creates two files:

  • ~/.ssh/id_ed25519 — your private key (NEVER share this)
  • ~/.ssh/id_ed25519.pub — your public key (copy this to servers)

3b. Copy your public key to each server:

# Replace with your actual server IPs:
ssh-copy-id deploy@192.168.56.10
ssh-copy-id deploy@192.168.56.11
ssh-copy-id deploy@192.168.56.12

You'll be asked for the deploy user's password once. After this, no more passwords!

3c. Test that passwordless SSH works:

ssh deploy@192.168.56.10    # Should log in WITHOUT asking for a password
ssh deploy@192.168.56.11
ssh deploy@192.168.56.12
🛑 Don't skip this step! If SSH key authentication is not set up, Ansible will fail with authentication errors. Every single server must accept your SSH key before running Ansible.

ansible/inventories/dev/hosts.yml

Just edit the numbers after ansible_host: to match your actual server IPs.

Save the file when done.

5 Set Up Your Vault (Secret Passwords)

The vault is an encrypted file that stores sensitive passwords. Ansible reads it automatically during runs, but only you can open it with your vault password.

5a. Create the vault secrets file from the example template:

# From inside the ansible/ directory:
cd /path/to/cmdb/ansible

cp vault/vault_secrets.yml.example vault/vault_secrets.yml

5b. Edit the file and fill in real passwords:

# Open vault/vault_secrets.yml in your editor and change CHANGE_ME values:
---
vault_glassfish_admin_password: "MyGlassFishPass123!"
vault_mongodb_admin_password:   "MyMongoAdminPass456!"
vault_mongodb_app_password:     "MyMongoAppPass789!"
vault_jenkins_admin_password:   "MyJenkinsPass321!"
⚠️ Password rules: Use strong passwords (8+ characters, mix of letters, numbers, symbols). Avoid @ in MongoDB passwords as it can break connection strings.

5c. Encrypt the vault file:

ansible-vault encrypt vault/vault_secrets.yml

You'll be asked to create a vault password. Choose something memorable — you'll need it every time you run a playbook. The file will become unreadable gibberish (that's good!).

5d. Create a vault password file (so you don't type it every run):

# Create a secure file to hold your vault password:
mkdir -p ~/.ansible
echo "YourVaultPasswordHere" > ~/.ansible/vault_pass_cmdb
chmod 600 ~/.ansible/vault_pass_cmdb
🛑 Critical security note: Never commit vault_secrets.yml (unencrypted) or ~/.ansible/vault_pass_cmdb to Git. The vault_secrets.yml in the repo should always be the encrypted version.

5e. Uncomment the vault password line in ansible.cfg:

Open ansible/ansible.cfg and find this line (near line 26). Remove the # at the beginning:

# Before:
# vault_password_file = ~/.ansible/vault_pass_cmdb

# After:
vault_password_file = ~/.ansible/vault_pass_cmdb

6 Install Required Ansible Collections

Your project uses extra Ansible "plugins" called collections. Install them with one command:

# Run this from inside the ansible/ folder:
cd /path/to/cmdb/ansible

ansible-galaxy collection install -r requirements.yml

This installs:

  • ansible.posix — for firewall and SELinux tasks
  • community.general — for timezone and other utilities
  • community.mongodb — for MongoDB management
✅ You only need to run this once (unless you update requirements.yml).

7 Test the Connection to All Servers

Before running any playbooks, verify Ansible can reach all your servers:

# From inside the ansible/ folder:
cd /path/to/cmdb/ansible

ansible all -i inventories/dev/hosts.yml -m ping

✅ Expected output (SUCCESS):

cmdb-dev-app01 | SUCCESS => {
    "changed": false,
    "ping": "pong"
}
cmdb-dev-db01 | SUCCESS => {
    "changed": false,
    "ping": "pong"
}
cmdb-dev-ci01 | SUCCESS => {
    "changed": false,
    "ping": "pong"
}

❌ If you see UNREACHABLE errors:

  • Check the server IPs in hosts.yml are correct
  • Make sure SSH key authentication works (re-do Step 3)
  • Check the servers are running and reachable from your network
  • Check that firewall on servers allows port 22 (SSH)

8 Run the Full Provisioning Playbook

This is the big moment! The master playbook (site.yml) will set up ALL three servers — Java, MongoDB, GlassFish, and Jenkins — in one go.

⚠️ This takes a while! Expect 10–30 minutes on the first run. It downloads and installs a lot of software. Be patient and watch the output. 
# From inside the ansible/ folder:
cd /path/to/cmdb/ansible

ansible-playbook -i inventories/dev/hosts.yml playbooks/site.yml

Ansible will go through each server and run tasks one by one. You'll see output like:

PLAY [Common baseline for all hosts] ****************************

TASK [common : Set hostname to inventory hostname] **************
changed: [cmdb-dev-app01]
changed: [cmdb-dev-db01]
changed: [cmdb-dev-ci01]

TASK [common : Ensure EPEL repository is present] ***************
changed: [cmdb-dev-app01]
...

PLAY RECAP ******************************************************
cmdb-dev-app01  : ok=25  changed=18  unreachable=0  failed=0
cmdb-dev-db01   : ok=22  changed=15  unreachable=0  failed=0
cmdb-dev-ci01   : ok=20  changed=14  unreachable=0  failed=0

Understanding the output colors/statuses:

StatusMeaning
okTask ran and nothing needed to change (already correct)
changedTask ran and made a change (installed something, edited a file, etc.)
failedTask failed — check the error message ❌
unreachableCould not connect to the server — SSH/network issue ❌
✅ Playbooks are safe to re-run! Ansible is idempotent — running it multiple times won't break anything. If a task already set things up correctly, it just shows "ok" and skips it. You can re-run the playbook to fix failed tasks.

Run individual parts instead of everything at once:

# Only set up infrastructure (common + Java) on all servers:
ansible-playbook -i inventories/dev/hosts.yml playbooks/infra.yml

# Only set up MongoDB:
ansible-playbook -i inventories/dev/hosts.yml playbooks/database.yml

# Only set up GlassFish:
ansible-playbook -i inventories/dev/hosts.yml playbooks/appserver.yml

# Only set up Jenkins:
ansible-playbook -i inventories/dev/hosts.yml playbooks/ci.yml

9 Build & Deploy the CMDB Application

Now that the infrastructure is ready, deploy the actual CMDB Java application.

9a. Build the WAR file with Maven (on your laptop):

# Go to the root of the cmdb project:
cd /path/to/cmdb

# Build the project (this compiles Java code and creates a .war file):
mvn clean package -DskipTests

# The WAR file will be at:
ls target/cmdb.war
⚠️ You need Maven installed for this. Install it with: brew install maven (macOS) or sudo dnf install -y maven (Linux).

9b. Deploy the WAR to GlassFish using Ansible:

# From inside the ansible/ folder:
cd /path/to/cmdb/ansible

ansible-playbook -i inventories/dev/hosts.yml playbooks/deploy.yml \
    -e "app_war_source=../target/cmdb.war"

This will copy the WAR file to the App Server and deploy it to GlassFish automatically.

🔧 Troubleshooting Common Errors

❌ "UNREACHABLE! Connection refused"

Cause: SSH not running or wrong IP.

  • Verify the server IP: ping 192.168.56.10
  • Check SSH is running on the server: systemctl status sshd
  • Make sure port 22 is open in the server's firewall

❌ "Permission denied (publickey)"

Cause: SSH key not copied to the server.

  • Re-run: ssh-copy-id deploy@<SERVER_IP>

❌ "AnsibleError: Attempting to decrypt but no vault secrets found"

Cause: Ansible can't find the vault password.

  • Make sure ~/.ansible/vault_pass_cmdb exists and has the right password
  • Uncomment vault_password_file in ansible.cfg

❌ "fatal: [host]: FAILED! => ...sudo: a password is required"

Cause: The deploy user can't run sudo without a password.

  • On the server, run: echo '%wheel ALL=(ALL) NOPASSWD: ALL' | sudo tee /etc/sudoers.d/wheel-nopasswd
  • Make sure the deploy user is in the wheel group: sudo usermod -aG wheel deploy

❌ "Error: Package mongodb-org not found"

Cause: MongoDB repo wasn't added correctly.

  • Run the database playbook with verbose output: add -vvv
  • Check the MongoDB repo file on the server: cat /etc/yum.repos.d/mongodb-org-7.0.repo

❌ GlassFish doesn't start (port 8080 not available)

Cause: Another service (like Tomcat or Jenkins) is using port 8080.

  • Check: sudo ss -tlnp | grep 8080
  • Jenkins also uses 8080 — make sure GlassFish and Jenkins are on different servers (they are in the default setup)

❌ "Missing required collections"

Cause: You skipped Step 6.

  • Run: ansible-galaxy collection install -r requirements.yml

Getting more details about an error:

# Add -vvv for very verbose debug output:
ansible-playbook -i inventories/dev/hosts.yml playbooks/site.yml -vvv 2>&1 | tee ansible-run.log
